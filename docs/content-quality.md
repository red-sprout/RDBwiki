# Content Quality Guide

RDB Wiki의 문서는 단순 학습용 요약이 아니라 실제 DB 운영자, DBA, 백엔드 개발자, 인프라 운영자가 참고할 수 있는 DBMS 레퍼런스 문서로 작성합니다.

## 금지 표현

문서 본문, 제목, 설명, UI 안내 문구에서 독자 수준을 나누는 다음 표현을 사용하지 않습니다.

- 초심자
- 입문자
- 초보자
- 주니어
- 시니어
- 전문가
- 고급자
- 비전공자

독자 수준을 분류하지 않습니다. 기능, 내부 동작, 운영 판단 기준, 장애 분석 절차를 기준으로 문서를 구성합니다.

## 문체

- 한국어 습니다체를 사용합니다.
- 설명은 명확하게 쓰되, 내용을 단순화해서 부정확하게 만들지 않습니다.
- 비유나 과도한 쉬운 설명을 피합니다.
- 실제 운영 판단에 필요한 공식 용어, 설정명, wait event, catalog/view 이름, 실행 계획 용어를 정확히 사용합니다.
- 공식문서에서 확인해야 할 키워드와 관련 설정을 명시합니다.

## 최소 품질 기준

초기 seed 문서와 새 문서는 최소한 다음 내용을 포함합니다.

- 본문 최소 1,500단어 또는 이에 준하는 한국어 장문 분량
- 최소 20개 이상의 의미 있는 섹션
- 한 줄 결론 1~2문장
- 적용 범위
- 핵심 개념
- 내부 동작 설명
- 해당 기능이 등장한 배경 또는 운영상 필요성
- 장점과 이점 4개 이상
- 단점과 운영 리스크 6개 이상
- 사용해도 되는 조건과 피해야 하는 조건
- 하이브리드 설계 패턴 또는 대체 설계
- 마이그레이션/롤백/승격 기준
- DBMS별 차이 표 1개 이상
- MySQL, PostgreSQL, Oracle 각각의 동작 방식, 관련 설정, 진단 방법, 운영 주의점
- 실무 판단 기준 7개 이상
- 장애/성능 이슈 패턴 5개 이상
- 진단 체크리스트 7개 이상
- 예시 SQL 또는 확인 쿼리 3개 이상
- 공식문서 참고 항목 3개 이상

분량 기준은 형식 채우기가 아니라 설명 밀도 기준입니다. 예를 들어 JSON/JSONB 문서라면 JSON 컬럼이 등장한 배경, DBMS별 지원 방식, 스키마 유연성, 반정규화 조회, 외부 API 원본 보관, 이벤트 payload, 무결성 부재, 인덱싱 비용, JOIN/집계 어려움, 복제 부하, 부분 업데이트 비용, 크기 제한, 하이브리드 패턴, 정규 컬럼 승격 절차까지 설명해야 합니다.

## 피해야 할 스타일

- 개념만 나열하는 문서
- 면접 답변처럼 짧은 문서
- 블로그식 감상문
- 비유 중심 설명
- DBMS별 차이를 지원 여부만으로 정리하는 표
- 공식문서 없이 추측으로 작성한 내용
- 독자 수준을 나누는 문장

## 권장 스타일

- 공식문서 기준의 용어를 사용합니다.
- DBMS별 구현 차이를 운영 영향과 연결합니다.
- 기능의 장점, 제약, 부작용, 장애 포인트를 함께 작성합니다.
- 진단 쿼리와 확인 절차를 포함합니다.
- 설정값과 파라미터의 의미를 설명합니다.
- trade-off를 명시합니다.
- 어떤 조건에서 유용한지와 어떤 조건에서 위험한지를 함께 설명합니다.
